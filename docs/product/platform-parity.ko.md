# 플랫폼 일관성 {#platform-parity}

Lava의 플랫폼 일관성 시스템은 어떤 제품 약속이 iOS, Android, 그리고 앞으로
나올 클라이언트에서 공통으로 지켜지는지를 정리해요. 이것은 기능 동작에 대한
공개된 약속이에요. 어디에서나 같은 의미여야 하는 것은 무엇인지, 의도적으로
각 OS에 맞춘 부분은 무엇인지, 아직 약속하지 않은 부분은 무엇인지를 보여줘요.

일관성 문서는 구현 계획이나 테스트를 대신하지 않아요.

- `lavasec-doc`은 제품과 동작에 대한 약속을 담아요.
- 내부 계획은 진행 상태, 순서, 비공개 위험 요소, 보드 동기화를 담아요.
- 플랫폼 저장소는 동작을 증명하는 코드, 픽스처, 테스트를 담아요.

문서와 출시된 코드가 다를 때는 문서가 갱신되기 전까지 코드가 기준이 돼요.
계획과 이 페이지가 다를 때는, 이 페이지를 제품 약속으로, 계획은 작업
대기열로 보면 돼요.

## 상태 용어 {#status-vocabulary}

| 상태 | 의미 |
|---|---|
| **출시됨** | 해당 플랫폼의 운영 코드에 구현되어 있어요. |
| **부분 구현** | 일부 동작은 있지만, 공개된 약속을 완전히 충족하지는 못해요. |
| **계획됨** | 플랫폼 약속에 포함되었지만, 아직 구현되지 않았어요. |
| **보류됨** | 유효한 기능이지만, 다음 플랫폼 마일스톤에는 필요하지 않아요. |
| **OS 맞춤형** | 사용자에게 주는 약속은 같지만, OS별로 구현 방식이 달라요. |
| **해당 없음** | 그 플랫폼에는 같은 기능이 존재할 필요가 없어요. |
| **제외됨** | 이전에 검토하거나 만들었지만, 의도적으로 제거했어요. |

## 기능 기록 형식 {#feature-record-format}

일관성 추적 대상 기능에는 항상 변하지 않는 기능 id가 있어야 해요. UI 문구가
바뀌어도 그대로 유지되는 `area.capability` 형태의 이름을 쓰세요. 예를 들어
`filtering.guardrail-precedence`나 `dns.encrypted-transports`처럼요.

완전한 기능 기록은 다음을 담고 있어요.

| 항목 | 용도 |
|---|---|
| `feature_id` | 계획, PR, 테스트, 문서에서 쓰는 변하지 않는 id예요. |
| 제품 약속 | 사용자가 믿고 의지할 수 있는 내용을, 플랫폼과 무관한 표현으로 적어요. |
| 일관성 요건 | Android가 iOS와 정확히 같아야 하는지, 의도 기준으로 맞추면 되는지, 의도적으로 달라도 되는지를 정해요. |
| 플랫폼 상태 | iOS, Android, 그리고 앞으로 나올 클라이언트의 상태예요. |
| 검증 방식 | 동작이 약속대로 유지되도록 하는 테스트, 픽스처, 소스 파일, 리뷰 체크예요. |
| 플랫폼 참고 | 나중에 다시 발견하지 않도록 명확히 적어 둔 OS별 차이점이에요. |

## 갱신 절차 {#update-workflow}

1. 변경 사항이 제품 약속, 개인정보 관련 약속, 요금제 경계, 또는 플랫폼 간
   동작을 바꿀 때 기능 id를 추가하거나 갱신하세요.
2. 작업이 필요할 때 구현 계획에서 같은 기능 id를 연결하세요.
3. 일치해야 하는 동작에 대해 플랫폼 테스트나 골든 픽스처를 추가하거나
   갱신하세요.
4. 한 플랫폼에서 동작이 출시되면 여기 상태를 갱신하고, 관련 기능 또는
   아키텍처 페이지를 함께 갱신하세요.
5. 구현 전용, 비공개, 가격, 법적 위험, 운영 관련 내부 세부 사항은 비공개로
   유지하고, 여기에는 공개된 약속만 정리하세요.

## 현재 일관성 대장 {#current-parity-ledger}

| 기능 id | 제품 약속 | iOS | Android | 일관성 요건 | 검증 / 소스 |
|---|---|---:|---:|---|---|
| `protection.local-dns-filtering` | Lava는 기기에서 직접 DNS를 필터링하며, 브라우징을 Lava 서버로 우회시키지 않아요. | 출시됨 | 계획됨 | 의도 기준으로 맞춤. OS 터널 API는 서로 달라요. | iOS 패킷 터널 아키텍처. Android `VpnService` 계획. |
| `protection.vpn-disclosure` | 앱은 OS가 로컬 DNS 필터링을 VPN이라고 부르는 이유를, VPN 권한/설정을 요청하기 전에 설명해요. | 출시됨 | 계획됨 | OS 맞춤형 문구와 권한 흐름. | 온보딩 문서. Android Play 고지 계획. |
| `filtering.guardrail-precedence` | 항상 켜진 가드레일이 사용자 허용 목록보다 우선해요. 유료 상태가 가드레일을 건너뛰는 일은 없어요. | 출시됨 | 계획됨 | 동작이 정확히 일치. | `CompactFilterSnapshotTests`. 이식 후 Android `FilterSnapshotTest`. |
| `filtering.source-url-only-catalog` | Lava는 카탈로그 메타데이터와 원본 소스 URL을 공개하며, 제3자 차단 목록 데이터 자체는 공개하지 않아요. | 출시됨 | 계획됨 | 개인정보/지식재산 모델이 정확히 일치. | 카탈로그 아키텍처. GPL/소스 URL 전용 법무 문서. |
| `filtering.on-device-parsing` | 선택한 목록은 기기에서 가져와 분석해요. 일상적인 도메인 기록은 Lava로 업로드하지 않아요. | 출시됨 | 계획됨 | 개인정보 보호가 정확히 일치, 네이티브 저장은 허용. | `BlocklistParserTests`. 이식 후 Android 파서 일관성 테스트. |
| `filtering.rule-budget` | 필터 한도는 임의의 목록 개수가 아니라, 컴파일된 규칙 수와 기기 안정성을 기준으로 정해요. | 출시됨 | 계획됨 | 사용자에게 보이는 모델은 동일. 플랫폼 메모리 한도는 다를 수 있어요. | iOS 필터 한도 테스트. 기기 한도 파악 후 Android 한도 테스트. |
| `dns.built-in-resolvers` | 사용자는 허용된 조회를 Lava로 보내지 않으면서 내장 리졸버 프리셋을 고를 수 있어요. | 출시됨 | 계획됨 | 리졸버 정책은 동일. 프리셋 구성은 단계적으로 출시될 수 있어요. | 리졸버 프리셋 테스트. 이식 후 Android 리졸버 DTO 테스트. |
| `dns.encrypted-transports` | 허용된 쿼리에 대해 암호화된 상위 DNS를 사용할 수 있어요. | 출시됨 | 계획됨 | 단계적 일관성 허용. Android v1은 DoT/DoQ보다 DoH로 먼저 시작할 수 있어요. | iOS 전송 테스트. Android 리졸버 테스트와 기기 QA. |
| `reports.local-only-diagnostics` | 리포트와 진단 정보는, 사용자가 직접 지원 번들을 보내지 않는 한 기기에만 머물러요. | 출시됨 | 계획됨 | 개인정보 보호가 정확히 일치. UI는 다를 수 있어요. | 버그 리포트 번들 테스트. 제작 후 Android 디버그 리포트 미리보기 테스트. |
| `account.optional-sign-in` | 보호 기능은 계정 없이도 작동해요. 로그인은 선택 사항이에요. | 출시됨 | 보류됨 | Android가 계정 기능을 공개하기 전에 제품 약속이 정확히 일치. | 계정 인증 문서. Android 온보딩/설정 리뷰. |
| `backup.zero-knowledge-settings` | 선택 사항인 설정 백업은 암호문만 저장해요. Lava는 백업 내용의 평문을 읽을 수 없어요. | 출시됨 | 보류됨 | Android가 백업을 제공하기 전에 개인정보 보호가 정확히 일치. | 제로 지식 백업 테스트. 제작 후 Android 암호화 일관성 테스트. |
| `plus.customization-boundary` | 무료 보호 기능은 계속 유용하게 쓸 수 있어요. Plus는 고급 맞춤 설정을 열어 주며, 가드레일 안전성은 절대 바꾸지 않아요. | 출시됨 | 계획됨 | 제품 경계는 동일. 스토어 구현은 OS 맞춤형이에요. | 구독 정책 테스트. 제작 후 Play Billing 권한 테스트. |
| `design.calm-earned-depth` | 기본 UX는 차분하며, 더 깊은 기술 화면이나 축하 화면은 그럴 만할 때나 요청할 때만 보여요. | 부분 구현 | 계획됨 | 공유 토큰/역할을 통해 디자인 의도 기준으로 맞춤. | 디자인 시스템 문서와 이식성 기반 계획. |
| `platform.ambient-presence` | OS가 네이티브 상시 표시 영역을 지원하면, 보호 상태를 앱 밖에서도 볼 수 있어요. | OS 맞춤형 | 계획됨 | 표시 영역이 아니라 의도 기준으로 일치. | iOS Live Activity 문서. Android 알림/빠른 설정 결정 대기 중. |

## Android 준비 점검 활용 {#android-readiness-use}

Android 구현을 시작하기 전에, 이 페이지를 Android 계획과 디자인 시스템
이식성 계획과 함께 검토해야 해요. 최소한의 Android 준비 약속은 다음과
같아요.

- 개인정보와 관련된 모든 기능에 기능 id가 있어야 해요.
- 정확히 일치해야 하는 동작에는 확인된 iOS 테스트나 픽스처 소스가 있어야
  해요.
- OS 맞춤형 동작에는 명확한 Android 입장이 있어야 해요.
- 보류된 기능은 이름을 적어 두어, Android MVP가 그 기능을 출시한다는 인상을
  실수로 주지 않도록 해야 해요.

이 검토는 구현 계획이나 리뷰 메모에 들어가야 하고, 이 페이지는 공개되고
오래가는 약속을 담아요.
