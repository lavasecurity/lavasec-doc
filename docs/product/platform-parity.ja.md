# プラットフォーム間の足並み {#platform-parity}

Lava のプラットフォーム間の足並み（パリティ）の仕組みは、どの製品上の約束が
iOS、Android、そして今後のクライアント間で共有されるのかを管理しています。これは機能の
ふるまいに関する公開された取り決めです。どこでも同じ意味でなければならないもの、あえて
プラットフォームごとのネイティブな作りにしているもの、そしてまだ約束していないものを示します。

このパリティのドキュメントは、実装計画やテストの代わりになるものではありません。

- `lavasec-doc` は製品とふるまいの取り決めを担います。
- 社内の計画は、リリースの状況、進める順番、社内向けのリスク、
  ボードとの同期を担います。
- プラットフォームのリポジトリは、ふるまいを裏づけるコード、フィクスチャ、テストを担います。

ドキュメントとリリース済みのコードが食い違っている場合は、ドキュメントが更新されるまではコードが正です。
計画とこのページが食い違っている場合は、このページを製品の取り決めとして扱い、
計画は作業のキューとして扱ってください。

## ステータスの語彙 {#status-vocabulary}

| ステータス | 意味 |
|---|---|
| **Shipped** | そのプラットフォームの本番コードに実装済み。 |
| **Partial** | 一部のふるまいは存在するが、公開された取り決めを完全には満たしていない。 |
| **Planned** | プラットフォームの取り決めの一部として受け入れ済みだが、まだ実装されていない。 |
| **Deferred** | 有効な機能ではあるが、次のプラットフォームのマイルストーンには必須ではない。 |
| **Platform-native** | ユーザーへの約束は同じだが、OS ごとの実装が異なる。 |
| **Not applicable** | そのプラットフォームには同等の機能を用意すべきではない。 |
| **Dropped** | 以前は検討または構築されたが、その後あえて取り除かれた。 |

## 機能レコードの書式 {#feature-record-format}

パリティ管理の対象となる機能には、安定した機能 id を持たせるべきです。UI の文言が
変わっても残るように、`area.capability` の形式の名前を使ってください。たとえば
`filtering.guardrail-precedence` や `dns.encrypted-transports` のように。

完全な機能レコードは、次の項目に答えます。

| 項目 | 目的 |
|---|---|
| `feature_id` | 計画、PR、テスト、ドキュメントで使う安定した id。 |
| 製品上の約束 | ユーザーが頼りにできること。プラットフォームに依存しない言葉で。 |
| パリティ要件 | Android が iOS と完全に一致すべきか、意図のうえで一致すべきか、あえて違ってよいか。 |
| プラットフォームのステータス | iOS、Android、今後のクライアントの状態。 |
| 担保 | ふるまいを正しく保つテスト、フィクスチャ、ソースファイル、レビューでのチェック。 |
| プラットフォームのメモ | あとから再発見するのではなく、明示しておくべき OS ごとの違い。 |

## 更新の進め方 {#update-workflow}

1. 製品上の約束、プライバシーに関する主張、プランの境界、プラットフォームをまたぐふるまいを
   変える変更があれば、機能 id を追加または更新する。
2. 作業が必要なときは、実装計画から同じ機能 id をリンクする。
3. 一致しなければならないふるまいについては、プラットフォームのテストやゴールデンフィクスチャを追加または更新する。
4. あるプラットフォームがそのふるまいをリリースしたら、ここのステータスを更新し、
   関連する機能ページやアーキテクチャのページも更新する。
5. 実装だけの話、社内向け、価格、法的リスク、運用上の社内の詳細は
   非公開のままにし、ここには公開する取り決めだけをまとめる。

| `filtering.guardrail-precedence` | 常時オンのガードレールはユーザーの許可リストより優先されます。有料の状態でガードレールを回避することは決してありません。 | Shipped | Planned | ふるまいを完全に一致。 | `CompactFilterSnapshotTests`。移植時の Android の `FilterSnapshotTest`。 |

| `filtering.rule-budget` | フィルターの上限は、任意のリスト数ではなく、コンパイル済みのルール数と端末の安全性に基づきます。 | Shipped | Planned | ユーザーに見えるモデルは同じ。プラットフォームのメモリ上限は異なる場合あり。 | iOS のフィルター上限テスト。端末の上限が判明したら Android の上限テスト。 |

| `account.optional-sign-in` | 保護はアカウントなしで動きます。サインインは任意です。 | Shipped | Deferred | Android がアカウント機能を出す前に、製品上の約束を完全に一致。 | アカウント認証のドキュメント。Android のオンボーディング / 設定レビュー。 |

| `platform.ambient-presence` | OS がネイティブなアンビエント面に対応している場合、保護のステータスをアプリの外にも表示できます。 | Platform-native | Planned | 面そのものではなく、意図のうえでのパリティ。 | iOS の Live Activity のドキュメント。Android の通知 / クイック設定の判断は保留中。 |

## Android 対応への活用 {#android-readiness-use}

Android の実装を始める前に、このページは Android の計画と、デザインシステムの
移植性計画と並べて見直すべきです。Android 対応の最低限の
取り決めは次のとおりです。

- プライバシーに関わるすべての機能に機能 id があること。
- 完全一致のふるまいには、対応する iOS のテストやフィクスチャのソースが特定されていること。
- プラットフォームごとのネイティブなふるまいには、明示的な Android のスタンスがあること。
- 先送りした機能には名前を付けて、Android MVP がそれらをリリースすると
  うっかり示唆しないようにすること。

この見直しは実装計画やレビューのメモに属するもので、このページは
公開された、長く保たれる取り決めを保ち続けます。
