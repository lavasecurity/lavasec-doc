# 주요 설계 결정

> 대상 독자: 엔지니어와 리더십. 이 문서는 Lava Security를 떠받치는 핵심 설계 결정을 ADR 형식으로 기록한 것이에요. 아키텍처, 프라이버시 약속, 제품 경계를 결정지은 결정, 그중에서도 시도했다가 되돌린 결정을 특히 담았어요. 각 항목은 **결정**, 그 **배경**, **근거**, 그리고 프로젝트 상태 범례에서 가져온 **상태**(채택 / 철회 / 대체 / 제안)를 제공해요.
>
> **코드가 우선이에요.** 계획과 실제 출시된 코드가 어긋날 때, 이 기록은 코드를 따르며 그 차이를 본문에서 짚어요.

**상태 범례(문서 세트의 상태 레인에 매핑):**

| 여기서의 상태 | 문서 세트 레인 의미 |
|---|---|
| **채택** | 구현됨 — 출시되었고 코드에서 확인됨 |
| **철회** | 폐기됨 — 만들었다가 제거/되돌림 |
| **대체** | 앞선 결정이 이후 결정으로 교체됨 |
| **제안** | 계획됨 — 설계/권고/기록되었으나 아직 이 트리에는 적용되지 않음 |

관련 읽을거리: 카탈로그 배포 모델은 [`../legal/gpl-source-url-only-compliance-decision.md`](../legal/gpl-source-url-only-compliance-decision.md)와 [`../legal/open-source-list-data-terms-carveout.md`](../legal/open-source-list-data-terms-carveout.md)에, 출시된 동작은 [`../product/features.md`](../product/features.md)에 있어요. 앞으로의 방향은 내부 로드맵에 있어요.

## 1. `NEPacketTunnelProvider`를 통한 기기 내 DNS 필터링

**결정.** `NEDNSProxyProvider`, `NEFilterProvider`, `NEDNSSettingsManager`, Safari 콘텐츠 차단기 대신, `NEPacketTunnelProvider` 패킷 터널(`LavaSecTunnel`, `com.lavasec.app.tunnel`)을 통해 DNS를 **기기에서 직접** 필터링해요.

**배경.** 이 제품은 비기술 사용자(부모, 어르신)를 위한 프라이버시 우선 필터로, 계정 없이 소비자용 App Store를 통해 배포돼요. 경쟁하는 NetworkExtension 프로바이더와 관리형 DNS API는 감독/MDM 관리 기기로 제한되거나 앱의 모든 DNS를 다루지 못하고, 리졸버 측 모델은 사용자의 도메인 흐름을 기기 밖으로 보내게 돼요.

**근거.** 패킷 터널은 (a) 관리되지 않는 소비자 기기에서 작동하고 (b) 모든 DNS 판단이 기기에서 일어나게 하는 유일한 프로바이더예요. 이것이 프라이버시 약속의 토대예요. *모든 DNS 필터링은 기기에서 이루어지고, Lava는 사용자의 탐색을 자사 서버로 절대 보내지 않으며 방문하는 도메인 흐름을 절대 받지 않아요.* 대신 받아들인 절충점은 터널이 지켜야 하는 iOS **확장당 약 50 MiB 메모리 상한**이에요. 이 제약은 아래 여러 후속 결정의 형태를 좌우해요.

**상태.** **채택**(기반 결정. 초기 프로토타입부터 코드에 존재).

## 2. source-url만으로 하는 블록리스트 배포

**결정.** Lava는 상위 블록리스트의 **URL과 허용 해시**만 게시하고, 기기는 각 `source_url`에서 리스트 **바이트**를 직접 내려받은 뒤 로컬에서 파싱, 정규화, 중복 제거, 필터링을 해요. Lava는 서드파티 블록리스트 바이트를 **절대** 저장, 미러링, 변환, 제공하지 않아요. Worker는 카탈로그 **메타데이터** JSON만 R2에 쓰며(`raw_r2_key`/`normalized_r2_key`는 null), 

**배경.** 이전 설계는 자문 검토를 위해 원본 블록리스트 바이트를 R2에 미러링했어요. 다수의 상위 리스트(HaGeZi, OISD)는 GPL-3.0이므로, 그 바이트를 호스팅하면 Lava가 GPL 데이터의 재배포자가 돼요.

**근거.** Lava를 블록리스트 배포자가 아니라 로컬 필터링 엔진 / 사용자 에이전트로 다루면 GPLv3 재배포와 App Review 노출을 최소화할 수 있어요. 기기는 내려받은 바이트를 카탈로그의 `accepted_source_hashes`로 검증하고, 불일치 시 마지막 정상 캐시로 폴백하거나 차단된 상태로 안전하게 실패해서, 미러 파이프라인이 제공하던 안전 속성을 회복해요. 또한 파싱된 모든 규칙 집합은 보호 도메인 필터를 거쳐, 상위 리스트가 Lava/Apple/신원 제공자 도메인을 차단하지 못하게 해요. 이 모델은 CI의 `check-gpl-blocklist-distribution.sh`로 강제돼요(미러 코드 없음, Lava 호스팅 아티팩트 URL 없음, GPL 소스의 기본 활성화 없음, R2 바이트 쓰기 없음).

**상태.** **채택**, 그리고 폐기된 R2 원본 미러 계획을 **대체**했어요(`plans/implemented/2026-05-25-gpl-raw-r2-blocklist-compliance-plan.md`, 헤더 "Superseded by the source-url-only implementation"). [`../legal/gpl-source-url-only-compliance-decision.md`](../legal/gpl-source-url-only-compliance-decision.md)를 참고하세요.

## 3. 암호화 리졸버 전송(DoH / DoH3 / DoT / DoQ)

**결정.** 평문 DNS와 기기 DNS 폴백에 더해, LavaSecCore로 분리한 네 가지 암호화 상위 전송을 제공해요. **DoH**(URLSession), **DoH3**(HTTP/3를 선호하는 DoH), **DoT**(풀링된 `NWConnection`, 엔드포인트당 최대 4개, 유휴-노후 갱신 및 1회 새 연결 재시도 포함), **DoQ**(DNS-over-QUIC)예요. 라우팅, 평문 DNS 강등, 백오프 게이트를 둔 엔드포인트별 페일오버, 기기 DNS 폴백은 `ResolverOrchestrator`에 있어요.

**배경.** 차단되지 않은 쿼리를 평문으로 리졸버에 전달하면, 기기 내 모델이 보호하려는 바로 그 도메인 흐름이 새어 나가요. 전송은 점진적으로 구축됐어요(DoH → DoH3 → DoT → DoQ).

**근거.** 암호화된 상위 전송은 차단되지 않은 쿼리를 종단 간으로 비공개로 유지해요. **DoH3**는 순전히 관찰 기반으로 표기돼요 — `assumesHTTP3Capable=true`를 설정하고 협상된 프로토콜을 관찰하며, **실제로 h3 협상이 관찰될 때만** UI가 `DoH3`(슬래시 없이)로 주석을 달아요. 약속하지는 않아요. h3는 연결마다 best-effort이고, 고정된 주장은 UDP를 막는 방화벽 뒤에서 실제 동작을 과장하게 되니까요. 유휴 갱신을 둔 DoT 풀링은 Cloudflare가 유휴 DoT 연결을 조용히 닫던 문제를 곧바로 해결한 것이에요.

**상태.** **채택**(네 전송 모두 존재하고 연결됨).

## 4. DoQ 연결 재사용 — 만들고, 기기에서 테스트하고, 철회

**결정.** DoQ에서 QUIC 연결을 재사용하지 **않아요**. `DoQTransport`는 **쿼리마다 새 QUIC 연결**을 열고, 4-레인 풀은 핸드셰이크 재사용이 아니라 동시성을 제공해요.

**배경.** RFC 9250은 각 DNS 쿼리를 자체 QUIC 스트림에 매핑하므로, 진짜 재사용에는 멀티 스트림 `NWConnectionGroup`/`openStream` API가 필요한데 이는 **iOS 26.0+ 전용**이고, 배포 하한은 iOS 17이에요. 그럼에도 iOS 26으로 게이트한 재사용 경로를 구현해(Xcode 26 SDK로 Debug+Release 컴파일) AdGuard DoQ를 상대로 **iOS 26.5 기기에서 테스트**했어요.

**근거.** 재사용 경로는 기기에서 시도할 때마다 실패했고(`openStream`/`receive` 오류 후 폴백이 "Socket is not connected"에 부딪힘), 쿼리당 기준선보다 **오히려 더 나쁜** 결과를 냈어요(대조: 핸드셰이크 34회 / 쿼리 35회, 전부 성공). 이는 "새 Network 프레임워크로 QUIC는 보류하라"는 Apple DTS 지침을 실증적으로 확인해 줘서, 작업을 출시하지 않고 되돌렸어요. API가 성숙하기 전 재시도를 막기 위해 문서와 가드 테스트 근거에만 이 결과를 남겨 뒀어요.

**상태.** **철회**(배포 하한이 iOS 26에 도달할 때까지 보류). DoQ는 쿼리당 새 연결로 설명하세요.

## 5. 통합 `DNSResolvingTransport` 프로토콜 거부

**결정.** 리졸버 전송을 하나의 `DNSResolvingTransport` 프로토콜로 통합하지 **않고**, 클로저 기반의 `ResolverOrchestrator.Executors` 이음매를 유지해요.

**배경.** 한 리팩터(이슈 407)가 모든 전송 위에 하나의 프로토콜을 두자고 제안했어요.

**근거.** 전송들은 너무 이질적이에요 — 비동기 암호화 실행기(DoH/DoT/DoQ)와 동기식 다중 주소 평문/기기 전송 — 그래서 통합 프로토콜은 기존의 주입 가능한 클로저 이음매보다 더 나쁜 추상화가 될 거예요. 이 이음매는 이미 와이어 실행을 테스트 가능하게 유지해요.

**상태.** **철회** / 구현 안 함(나쁜 추상화로 종료).

## 6. 영지식 암호화 백업(비밀번호 없음, 패스키 예외 명시)

**결정.** **최소화된** 설정 페이로드를 클라이언트 측에서 백업해요. AES-256-GCM이 무작위 32바이트 페이로드 키로 봉인하고, 그 키는 PBKDF2-HMAC-SHA256(프로덕션에서 **210,000**회 반복)을 통해 비밀별 **키 슬롯**에 래핑돼요. 암호문과 비밀이 아닌 메타데이터만 Supabase `user_backups` 테이블에 업로드돼요(사용자별 RLS). 출시된 흐름은 **비밀번호 없음** 방식이에요 — 기기 비밀 슬롯(기기 로컬 Keychain) + 보조 복구 슬롯 + 선택적 패스키 슬롯이에요.

**배경.** 선택적 계정 로그인(Apple + Google만)으로 기기 간 설정 복원을 할 수 있어요. 서버는 사용자의 블록리스트, 허용 목록, 리졸버 선택, 기타 설정을 절대 읽을 수 없어야 해요.

**근거.** 평문과 복호화 비밀은 기기에만 존재하고, 서버는 사용자당 불투명한 봉투 하나만 보유해요. 보조 복구는 의도적으로 2요소예요 — `SHA256("LavaSec assisted recovery v1\0" + serverRecoveryShare + "\0" + normalizedPhrase)`(NUL로 구분된 입력)은 서버가 보유한 share와 사용자의 8단어 복구 구문(약 105비트)을 **둘 다** 요구하므로, 어느 한쪽만으로는 복호화되지 않아요. 잠금 해제 자료는 기기 로컬(`kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`)에 저장되며, 동기화되는 iCloud Keychain에는 저장하지 **않아요** — 원래 계획의 동기화 설계를 뒤집은 프라이버시 강화예요. **패스키 슬롯도 진정으로 영지식**이에요. 클라이언트를 절대 떠나지 않는 WebAuthn **PRF / `hmac-secret`** 인증기 출력(HKDF-SHA256으로 파생)으로 래핑되므로, 서버가 보유한 어떤 값으로도 풀 수 없어요. 서비스 역할 패스키 테이블도, Worker WebAuthn 어서션 게이트도 없어요 — 앞선 서버 게이트 패스키 설계는 폐기되어 서버 측 패스키 상태가 모두 제거됐어요(`Sources/LavaSecCore/ZeroKnowledgeBackupEnvelope.swift`).

**상태.** **채택**(비밀번호 없는 모델, 보조 복구, 영지식 PRF 파생 패스키 슬롯 모두 코드에 존재). 패스키를 실제 기기에서 완전히 프로덕션 준비된 복구 요소로 만드는 것(PRF 모델용 Associated Domains / AASA 호스팅)은 **제안**(백로그) 상태예요.

## 7. 차단 우선 Connect-On-Demand

**결정.** OS가 멈춘 터널이 자동으로 다시 시작되도록 `NEOnDemandRuleConnect` 규칙을 추가하되, **차단 우선**을 안전한 기본값으로 둬요 — 재사용할 수 있는 필터 스냅샷이 없으면 터널은 트래픽을 필터 없이 통과시키지 않고 모두 차단해요. 온디맨드는 **어떤 중지보다 먼저 비활성화**되어 VPN을 끌 수 있게 유지돼요.

**배경.** iOS가 조용히 터널을 멈추고(이유 17) 약 45분 동안 아무것도 재시작하지 않아, 사용자가 보호받지 못하는 상태가 됐어요. 온디맨드를 순진하게 켜면 VPN을 끌 수 없게 되고, 통과 우선 기본값은 그 간극 동안 트래픽을 그대로 흘려보내요.

**근거.** 온디맨드는 조용한 중지 간극을 닫고, 중지 전 비활성화는 사용자가 보호를 끌 수 있는 능력을 지키며, 차단 우선은 그 간극이 조용히 필터 없는 상태가 아니라 안전하도록 보장해요(`reconcileTunnelSnapshotAfterLaunch`로 회복). 이 변경에는 부작용이 있었어요 — 온디맨드가 온보딩 중 "VPN 구성 추가" 시스템 프롬프트를 다시 띄웠어요 — 그래서 여러 커밋의 수정 체인을 낳았어요: 설치 시 온디맨드 활성화 중단, 온보딩 완료를 기준으로 실행/보호 복원 게이트, 그리고 **상속/고아 구성을 저장이 아니라 제거로 무력화**(`removeFromPreferences`, 조용히)였어요. `on-demand=false`로 저장하면(`saveToPreferences`) 프롬프트가 다시 떴거든요.

**상태.** **채택**(온디맨드 재시작과 온보딩/차단 우선 수정 체인).

## 8. 모듈형 VPN 리팩터와 발열 회귀 규율

**결정.** 캐시 우선 켜기, 제한된 병렬 페치, 플랩 병합을 위해 VPN 경로(VPNLifecycleController, ProtectionActionOrchestrator, ResolverOrchestrator, FilterArtifactStore, DNSResponseCache, RuleSetCache, FilterSnapshotPreparationService)를 재구성해요 — 배터리/지연을 명시적인 p50/p95 목표를 둔 제품 요구사항으로 다루고, **기기에서**(시뮬레이터가 아니라) 프로파일링해요.

**배경.** 켜기 / 새로고침 / 일시정지 / 재개가 느렸어요. 리팩터 도중 발열 회귀가 나타났어요(CPU 134%, 높은 에너지, 뜨거운 폰). 대규모 에이전트 패널이 회귀 이전 증거로 의심된 원인을 먼저 반박했고, 이어진 실기기 캡처가 그것을 확인했어요.

**근거.** 진짜 원인은 스스로 지속되는 `NEVPNStatusDidChange` 새로고침 루프였어요 — drop-reentrant 가드가 교체된 뒤 영원히 재무장하던 병합 루프(초당 약 370 이벤트, 메인 스레드 약 100%, `vpn-debug-log.jsonl`이 약 180–210 MB까지 커짐)예요. 수정은 캐시된 매니저 상태를 읽고 루프를 제한해요. 계획 자체의 전후 기기 아티팩트는 iPhone 15 Pro에서 웜 켜기(`action.turnOn`)가 **2,722 ms → 287 ms**로 떨어진 것을 기록해요. 별도의 후속 모듈화 이후 기회 검토에서는 같은 기기에서 웜 경로를 **112 ms**(디코드 51 + managerSetup 57)로 측정했어요. 이 사건은 기준을 세웠어요: 구조적 리팩터는 측정된 발열 회귀가 제한될 때까지 멈추고, 시뮬레이터의 열/배터리 결과는 무의미한 것으로 거부해요.

**상태.** **채택**(`plans/implemented/2026-06-12-modular-speed-up-plan.md`). 모듈화 이후 검토는 `PacketTunnelProvider`와 `AppViewModel`을 살아남은 알려진 갓 오브젝트로 남겨 둬요.

## 9. 리스트 개수 상한이 아닌 필터 규칙 예산

**결정.** 활성화된 리스트 개수가 아니라 **필터 규칙 예산**으로 티어를 게이트해요 — **Free 50만 / Plus 200만** 컴파일된 도메인 규칙이에요. 하드 **약 326만 규칙 기기 가드레일**(`maxResidentMegabytes 32.0`, `baselineMegabytes 4.0`, `estimatedBytesPerRule 9.0` → `maxFilterRuleCount = 3,262,236`)은 **모두에게** 적용되며 **결코 유료 장벽이 아니에요**. 컴팩트 도메인 블롭은 `mmap`(`.mappedIfSafe`)되어 파일 기반으로 유지되고 jetsam이 계산하는 `phys_footprint` 밖에 머물러요. 디코드된 엔트리 테이블만 상주 메모리를 차지해요.

**배경.** 예전 상한은 리스트 **개수**였어요(무료 3개 / 유료 10개). 한 리스트가 1천 개 또는 1백만 개 규칙을 담을 수 있으니, 개수는 실제로 제약되는 자원 — NE 50 MiB 메모리 상한 — 에 대한 정직하지 못한 대용물이었어요.

**근거.** 규칙은 실제 메모리에 매핑되므로, 들어맞는 리스트 조합은 무엇이든 허용돼요. 권위 있는 강제는 `FilterSnapshotPreparationService`에서 중복 제거된 합집합에 대해 컴파일 시점에 이뤄져요(기기 가드레일 먼저, 그다음 티어 한도). 선택 시점의 UI 미터는 1.10 소프트 상한 여유를 둔 리스트별 합계를 써요. 예산 초과 구성은 터널이 jetsam되게 두지 않고 결정론적으로 거부돼요(보호는 꺼진 채로 유지).

**상태.** 코드에서 **채택**(`SubscriptionPolicy.swift`)되었고, 리스트 개수 상한을 **대체**했어요. 이를 이끈 계획(`plans/under_review/2026-06-13-filter-rules-budget-tier-revamp.md`)은 아직 검토 중이고, 공개 사이트의 "활성 블록리스트 3 → 10" 문구는 **낡았어요** — 실제 게이트는 규칙 예산이에요. [`../product/features.md`](../product/features.md)를 참고하세요.

## 10. 마크다운 기반 계획 + 단방향 Linear 동기화

**결정.** `plans/<lane>/`의 마크다운 파일이 **단일 진실 공급원**이고, **레인 폴더가 권위 있는 상태**예요(`implemented`, `inflight`, `under_review`, `backlog`, `dropped`). `main`으로의 푸시는 계획을 Linear(팀 LAV)로 **단방향** 동기화하며, 생성 이후에는 제목/설명만 갱신해요. 별도의 **수동·검토된** 복귀 경로가 Linear의 상태/우선순위/레인을 계획 frontmatter로 다시 끌어와요.

**배경.** 소규모 팀에는 프로젝트 트래커와 다투지 않는, 도구에 구애받지 않고 검토 가능한 계획 상태가 필요하고, 자율 에이전트 루프에는 계획 상태를 읽고 쓸 안정적인 곳이 필요해요.

**근거.** 필드 소유권 분리는 두 시스템을 충돌 없이 유지해요 — 마크다운은 콘텐츠를, Linear는 분류 상태를 소유 — 그래서 푸시가 사람의 분류를 덮어쓰지 않아요. `dropped/` 레인은 취소된 계획을 동기화 파이프라인 밖에 두어 다시 나타나지 않게 해요(Allowed Exceptions Guardrails / LAV-5가 거부되면서 생성됨). 계획 안의 낡은 frontmatter는 상태가 아니라 문서 버그예요. 폴더가 우선이고, 코드가 "Backlog" frontmatter에도 불구하고 기능이 출시되었음을 보일 때(예: 계정 삭제)는 코드가 우선이에요.

**상태.** **채택**(`scripts/sync-plans-to-linear.mjs`, `.github/workflows/sync-plans.yml`; `dropped/` 레인 사용 중).

## 11. 리포 분할 + 클라이언트의 카피레프트 오픈소스화

**결정.** 모노레포를 컴포넌트별 리포(`lavasec-ios`, `-android`, `-web`, `-infra`, `-doc`, `-runner`)로 나누고, Mullvad/ProtonVPN의 카피레프트 선례에 따라 Apache-2.0 대신 **AGPL-3.0으로 자사 클라이언트를 오픈소스화**해요.

**배경.** 컴포넌트별 개발과 클라이언트 오픈소스화예요. 라이선스 쟁점은 경쟁사가 클라이언트를 포크해 비공개로 닫고 가격으로 깎아내릴 수 있느냐예요.

**근거.** 카피레프트는 파생물이 공개 상태를 유지하도록 강제해 클라이언트의 비공개 포크를 막아요 — "공개 클라이언트, 비공개 백엔드/운영" 태세로, 백엔드·법무·운영은 비공개로 유지해요. 평범한 GPL-3.0이 아니라 AGPL-3.0을 택한 건 네트워크 사용 허점을 막기 위해서예요. 알려진 GPL 대 App Store 배포 긴장은, Lava 자신이 자체 저작권으로 App Store 바이너리의 배포자가 됨으로써 처리돼요.

**상태.** **채택.** 리포 분할은 **완료**됐어요 — 각 컴포넌트가 자체 리포지토리에 있고, 공개 `lavasec-ios` 클라이언트는 태그 v0.4.0이며, Android, 마케팅 사이트, 백엔드/인프라, 문서, CI/릴리스 파이프라인은 각각 별도 리포지토리예요 — 그리고 `lavasec-ios`의 `README.md` "Repository layout" 섹션은 그 리포의 컴포넌트별 내용(`LavaSecApp/`, `LavaSecTunnel/`, `LavaSecWidget/`, `Shared/`, `Sources/`, `Tests/`)만 나열하고 인프라는 별도의 비공개 리포지토리에 있다고 표시해요. 클라이언트는 **AGPL-3.0**으로 오픈소스화됐어요 — `lavasec-ios`의 `LICENSE`는 GNU Affero General Public License v3이고 `README.md`에는 AGPL-3.0 배지가 있어요.

Darwin 알림은 **터널→앱** 방향으로만, 상태 변경 알림으로 남아 있어요. | **채택**(provider-message가 유일한 앱→터널 제어. Darwin은 터널→앱 상태용만) |
