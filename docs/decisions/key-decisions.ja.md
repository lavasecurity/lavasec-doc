## 1. `NEPacketTunnelProvider` によるオンデバイス DNS フィルタリング {#1-on-device-dns-filtering-via-nepackettunnelprovider}

**決定。** DNS を `NEDNSProxyProvider`、`NEFilterProvider`、`NEDNSSettingsManager`、Safari コンテンツブロッカーではなく、`NEPacketTunnelProvider` のパケットトンネル(`LavaSecTunnel`、`com.lavasec.app.tunnel`)を通じて **端末上でローカルに** フィルタリングする。

**背景。** この製品は、技術にくわしくない人(保護者や年配の方)向けの、プライバシーを最優先したフィルターで、一般向けの App Store で配布され、アカウント登録は不要です。競合となる NetworkExtension 系のプロバイダーや管理 DNS の API は、監視対象・MDM 管理下の端末に限られていたり、アプリの DNS をすべてカバーできなかったりします。さらに、リゾルバー側で処理するモデルだと、ユーザーが見たドメインの流れが端末の外へ出てしまいます。

**根拠。** パケットトンネルだけが、(a) 管理されていない一般ユーザーの端末でも動き、(b) すべての DNS 判定を端末上で行えます。これがプライバシーの約束の土台です。つまり、*DNS フィルタリングはすべて端末上で行われ、Lava はあなたの通信を自社サーバー経由で流すことは決してなく、あなたが訪れたドメインの流れを受け取ることもありません。* その代わりに受け入れたトレードオフが、トンネルが収まらなければならない iOS の **拡張機能あたり約 50 MiB のメモリ上限** です。この制約が、以下のいくつかの決定を形づくっています。

**ステータス。** **採用**(基盤となるもの。最初のプロトタイプからコードに入っている)。

## 2. ソース URL のみのブロックリスト配布 {#2-source-url-only-blocklist-distribution}

**決定。** Lava が公開するのは、上流のブロックリストの **URL と受け入れ可能なハッシュ** だけです。端末は各 `source_url` から直接リストの **バイト列** を取得し、ローカルで解析・正規化・重複排除・フィルタリングします。Lava はサードパーティのブロックリストのバイト列を保存・ミラー・変換・配信することは **一切ありません**。Worker が R2 に書き込むのはカタログの **メタデータ** JSON のみです(`raw_r2_key` / `normalized_r2_key` は null)。

**背景。** 以前の設計では、弁護士が配布内容を確認できるように、生のブロックリストのバイト列を R2 にミラーしていました。上流のリストの多く(HaGeZi、OISD)は GPL-3.0 なので、そのバイト列をホストすると Lava が GPL データの再配布者になってしまいます。

**根拠。** Lava をブロックリストの配布者ではなく、ローカルのフィルタリングエンジン / ユーザーエージェントとして扱うことで、GPLv3 の再配布や App Review のリスクを最小限にします。端末は各リストを、厳選された `source_url` から TLS で直接取得し、厳格なサイズ / ルール上限のもとでローカルに解析します。コミュニティのリストは配信されたとおりに受け入れられ(カタログの `accepted_source_hashes` は参考情報であって、厳格なゲートではありません — 単一の固定ハッシュでは速くローテーションする上流を追えず、誤った拒否しか生まなかった)、一方で Lava の脅威ガードレールのティアはハッシュ固定のままです。来歴はクライアントのハッシュゲートではなく、カタログで強制されます(`source_url` の変更は新しい `list_id` を使わなければなりません)。解析されたルールセットはどれも保護ドメインフィルターを通すので、上流のリストが Lava / Apple / ID プロバイダーのドメインをブロックすることはできません。このモデルは CI の `check-gpl-blocklist-distribution.sh` で強制されています(ミラーのコードなし、Lava がホストする成果物 URL なし、GPL ソースのデフォルト有効なし、R2 へのバイト書き込みなし)。

**ステータス。** **採用**。そして、これは見送られた R2 生ミラー案(`plans/implemented/2026-05-25-gpl-raw-r2-blocklist-compliance-plan.md`、見出し「Superseded by the source-url-only implementation」)を **置き換え** ました。[`../legal/gpl-source-url-only-compliance-decision.md`](../legal/gpl-source-url-only-compliance-decision.md) を参照。

## 3. 暗号化されたリゾルバートランスポート(DoH / DoH3 / DoT / DoQ) {#3-encrypted-resolver-transports-doh-doh3-dot-doq}

**決定。** 平文 DNS とデバイス DNS フォールバックに加えて、4 つの暗号化された上流トランスポートを LavaSecCore に切り出して出荷する。**DoH**(URLSession)、**DoH3**(HTTP/3 を優先する DoH)、**DoT**(プールされた `NWConnection`、エンドポイントあたり最大 4 本、アイドル古さの更新と新規接続による 1 回のリトライ付き)、**DoQ**(DNS-over-QUIC)。ルーティング、平文 DNS への降格、バックオフゲート付きのエンドポイントごとのフェイルオーバー、デバイス DNS フォールバックは `ResolverOrchestrator` にあります。

**背景。** ブロックされなかったクエリを平文でリゾルバーに転送すると、オンデバイスのモデルが守ろうとしているまさにそのドメインの流れが漏れてしまいます。トランスポートは段階的に作りました(DoH → DoH3 → DoT → DoQ)。

**根拠。** 上流を暗号化することで、ブロックされなかったクエリをエンドツーエンドで非公開に保ちます。**DoH3** のラベルは純粋に観測にもとづくものです。`assumesHTTP3Capable=true` を設定したうえでネゴシエートされたプロトコルを観測し、UI は h3 のネゴシエーションが **実際に観測されたときだけ** `DoH3`(スラッシュなし)と表示します。約束はしません。h3 は接続ごとのベストエフォートなので、固定的に主張すると UDP を遮断するファイアウォールの裏での挙動を誇張してしまうからです。アイドル更新付きの DoT プーリングは、Cloudflare がアイドルの DoT 接続を黙って閉じる問題への直接の対処でした。

**ステータス。** **採用**(4 つのトランスポートすべてが存在し、配線済み)。

## 4. DoQ の接続再利用 — 作って、実機で検証して、撤回 {#4-doq-connection-reuse-built-device-tested-reverted}

**決定。** DoQ で QUIC 接続を再利用 **しない**。`DoQTransport` は **クエリごとに新しい QUIC 接続** を開きます。4 レーンのプールが提供するのは並行性であって、ハンドシェイクの再利用ではありません。

**背景。** RFC 9250 は各 DNS クエリをそれぞれの QUIC ストリームに対応づけるので、本当の再利用にはマルチストリームの `NWConnectionGroup` / `openStream` API が必要です。これは **iOS 26.0 以降のみ** で、一方でサポートの下限は iOS 17 です。それでも iOS 26 ゲート付きの再利用パスは実装され(Xcode 26 SDK で Debug + Release ともにコンパイル)、AdGuard DoQ に対して **iOS 26.5 の実機で検証** されました。

**根拠。** 再利用パスは実機での試行すべてで失敗し(`openStream` / `receive` がエラー、その後フォールバックが「Socket is not connected」に当たる)、クエリごとのベースラインよりも **明確に悪い** 結果でした(対照群: 34 ハンドシェイク / 35 クエリ、全成功)。これは Apple DTS の「新しい Network フレームワークでの QUIC は当面見送るように」という案内を実証的に裏づけたので、この作業は出荷せず撤回しました。API が成熟する前に再挑戦しないよう、ドキュメントとガードテストの根拠にだけ、この知見を残しています。

**ステータス。** **撤回**(サポートの下限が iOS 26 に達するまで先送り)。DoQ はクエリごとの新規接続として説明すること。

## 5. 統一的な `DNSResolvingTransport` プロトコルを却下 {#5-reject-a-unifying-dnsresolvingtransport-protocol}

**決定。** リゾルバートランスポートを単一の `DNSResolvingTransport` プロトコルに統一 **しない**。クロージャーベースの `ResolverOrchestrator.Executors` の継ぎ目を維持する。

**背景。** あるリファクタリング(issue 407)が、すべてのトランスポートに 1 つのプロトコルを被せることを提案しました。

**根拠。** トランスポートはあまりに性質が違いすぎます。非同期の暗号化エグゼキューター(DoH / DoT / DoQ)に対して、同期的なマルチアドレスの平文 / デバイストランスポートがある、という具合です。だから統一プロトコルは、すでにワイヤー実行をテスト可能にしている、注入できるクロージャーの継ぎ目よりも、むしろ悪い抽象化になってしまいます。

**ステータス。** **撤回** / 実装しない(悪い抽象化としてクローズ)。

## 6. ゼロ知識の暗号化バックアップ(パスワードレス、パスキーの例外を明記) {#6-zero-knowledge-encrypted-backup-passwordless-passkey-exception-noted}

**決定。** **最小化** した設定ペイロードをクライアント側でバックアップする。AES-256-GCM がランダムな 32 バイトのペイロードキーで封をし、そのキーは PBKDF2-HMAC-SHA256(本番では **210,000** 回の反復)を介して、シークレットごとの **キースロット** に包まれます。アップロードされるのは暗号文と非シークレットのメタデータだけで、Supabase の `user_backups` テーブル(ユーザーごとの RLS)に入ります。出荷されたフローは **パスワードレス** です。デバイスシークレットスロット(デバイスローカルの Keychain)+ アシスト復元スロット + 任意のパスキースロット、という構成です。

**背景。** 任意のアカウントログイン(Apple + Google のみ)で、端末をまたいだ設定の復元ができます。サーバーは、ユーザーのブロックリスト、許可リスト、リゾルバーの選択、その他の設定を絶対に読めてはいけません。

**根拠。** 平文と復号用のシークレットは端末上にしか存在せず、サーバーが持つのはユーザーごとに 1 つの不透明なエンベロープだけです。アシスト復元はあえて二要素にしてあります。`SHA256("LavaSec assisted recovery v1\0" + serverRecoveryShare + "\0" + normalizedPhrase)`(NUL 区切りの入力)は、サーバーが持つシェアと、ユーザーの 8 単語のリカバリーコード(約 105 ビット)の **両方** を必要とするので、どちらか片方だけでは復号できません。ロック解除用のマテリアルはデバイスローカル(`kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`)に保存され、同期される iCloud キーチェーンには **入れません**。これは、元の案の同期する設計を覆したプライバシー強化です。**パスキースロットも本当にゼロ知識** です。クライアントから決して外に出ない WebAuthn の **PRF / `hmac-secret`** 認証器の出力(HKDF-SHA256 で導出)で包まれているので、サーバーが持つどんな値でもこれをほどくことはできません。サービスロールのパスキーテーブルも、Worker の WebAuthn アサーションゲートもありません。以前のサーバーゲート付きパスキー設計は取り下げられ、サーバー側のパスキー状態はすべて削除されました(`Sources/LavaSecCore/ZeroKnowledgeBackupEnvelope.swift`)。

**ステータス。** **採用**(パスワードレスモデル、アシスト復元、ゼロ知識の PRF 導出パスキースロット、すべてコードに入っている)。実機で、パスキーを完全に本番対応した復元可能な要素にすること(PRF モデル用の Associated Domains / AASA ホスティング)は **提案**(バックログ)です。

## 7. フェイルクローズな Connect-On-Demand {#7-fail-closed-connect-on-demand}

**決定。** OS に停止させられたトンネルが自動で再起動するよう `NEOnDemandRuleConnect` ルールを追加し、安全側のデフォルトとして **フェイルクローズ** にする。再利用できるフィルタースナップショットがないときは、トンネルはフィルターなしで通すのではなく、すべてのトラフィックをブロックします。On-demand は **どんな停止の前にも無効化** され、VPN がちゃんとオフにできる状態を保ちます。

**背景。** iOS がトンネルを黙って停止し(理由コード 17)、約 45 分間なにも再起動せず、ユーザーが無防備なままになっていました。素朴に on-demand を有効にすると VPN をオフにできなくなり、フェイルオープンなデフォルトだとその隙間でトラフィックを通してしまいます。

**根拠。** On-demand が黙って止まる隙間を塞ぎ、停止前の無効化がユーザーの保護をオフにする能力を守り、フェイルクローズがその隙間を、黙ってフィルターなしになるのではなく安全にします。`reconcileTunnelSnapshotAfterLaunch` で復旧します。この変更には副作用がありました。on-demand がオンボーディング中に「VPN 構成を追加」のシステムプロンプトを再び出してしまったのです。そこから複数コミットの修正連鎖が生まれました。インストール時に on-demand を有効化するのをやめ、起動 / 保護復元をオンボーディング完了に紐づけ、引き継がれた / 孤立した構成を `on-demand=false` を保存して(`saveToPreferences` はプロンプトを再表示してしまった)ではなく、**削除して無効化** する(`removeFromPreferences`、無音)ようにしました。

**ステータス。** **採用**(on-demand による再起動と、オンボーディング / フェイルクローズの修正連鎖)。

## 8. モジュラーな VPN リファクタリングと、発熱リグレッションの規律 {#8-modular-vpn-refactor-and-the-heat-regression-discipline}

**決定。** VPN のパス(VPNLifecycleController、ProtectionActionOrchestrator、ResolverOrchestrator、FilterArtifactStore、DNSResponseCache、RuleSetCache、FilterSnapshotPreparationService)を、キャッシュ優先のオン、上限付き並列フェッチ、フラップの合流のために再構成する。バッテリーとレイテンシーを、明示的な p50 / p95 の目標と **実機**(シミュレーターではない)でのプロファイリングを伴う、製品要件として扱います。

**背景。** オン / 更新 / 一時停止 / 再開が遅かったのです。リファクタリング中に発熱のリグレッションが現れました(CPU 134%、エネルギー High、本体が熱い)。大きなエージェントパネルが、まずリグレッション前の証拠を使って疑われた原因を否定し、その後の実機キャプチャでそれを確認しました。

**根拠。** 本当の原因は、自己持続する `NEVPNStatusDidChange` の更新ループでした。drop-reentrant ガードが置き換えられたあと、永遠に再武装し続ける合流ループ(約 370 イベント/秒、メインスレッド約 100%、`vpn-debug-log.jsonl` が約 180〜210 MB まで肥大化)になっていました。修正はキャッシュされたマネージャーの状態を読み、ループに上限を設けます。プラン自身の前後の実機成果物には、ウォームのオン(`action.turnOn`)が iPhone 15 Pro で **2,722 ms → 287 ms** に下がったことが記録されています。別の、あとのモジュラー化後の最適化機会レビューでは、同じ端末でウォームパスを **112 ms**(デコード 51 + managerSetup 57)と計測しました。この一件が基準を定めました。構造的なリファクタリングは、計測された発熱リグレッションに上限がつくまで止め、シミュレーターの熱 / バッテリー結果は無意味として却下する、というものです。

**ステータス。** **採用**(`plans/implemented/2026-06-12-modular-speed-up-plan.md`)。モジュラー化後のレビューは、`PacketTunnelProvider` と `AppViewModel` を、生き残った既知の god オブジェクトとして残しています。

## 9. リスト数の上限ではなく、フィルタールールの上限 {#9-filter-rules-budget-instead-of-a-list-count-cap}

**決定。** プランを **フィルタールールの上限** でゲートする。**無料 500K / Plus 2M** のコンパイル済みドメインルールで、有効リスト数では区切りません。ハードな **約 3.26M ルールのデバイスガードレール**(`maxResidentMegabytes 32.0`、`baselineMegabytes 4.0`、`estimatedBytesPerRule 9.0` → `maxFilterRuleCount = 3,262,236`)は **全員** に適用され、**決してペイウォールではありません**。コンパクトなドメインの blob は `mmap`(`.mappedIfSafe`)されるので、ファイルバックトされたまま、jetsam がカウントする `phys_footprint` の外に出ます。レジデントメモリを消費するのは、デコードされたエントリテーブルだけです。

**背景。** 古い上限はリストの **数**(無料 3 / 有料 10)でした。1 つのリストには 1K のルールも 1M のルールも入りうるので、数は本当に制約されているリソース、つまり NE の 50 MiB メモリ上限に対する不誠実な代理指標でした。

**根拠。** ルールは実際のメモリに対応するので、収まるならどんなリストの組み合わせでも許されます。権威ある強制は、`FilterSnapshotPreparationService` で重複排除した和集合に対してコンパイル時に走ります(まずデバイスガードレール、次にプラン上限)。選択時の UI メーターは、リストごとの合計に 1.10 のソフトシーリングのマージンを使います。上限超過の構成は(保護をオフのままにして)決定的に拒否され、トンネルを jetsam させることはありません。

**ステータス。** コード(`SubscriptionPolicy.swift`)で **採用** され、**v1.0.0** で出荷され、リスト数の上限を **置き換え** ました。ルールの上限が、いまや生きたプランのゲートです。ドメインごとの上限も 1.0 で引き上げられました(無料 25 / Plus 1,000 の、許可およびブロックするドメイン)。[`../product/features.md`](../product/features.md) を参照。

## 10. プランは markdown + 一方向の Linear 同期 {#10-plans-as-markdown-one-way-linear-sync}

**決定。** `plans/<lane>/` 内の markdown ファイルが **正(source of truth)** で、**レーンフォルダーが権威あるステータス**(`implemented`、`inflight`、`under_review`、`backlog`、`dropped`)です。`main` へのプッシュはプランを **一方向** で Linear(チーム LAV)に同期し、作成後はタイトル / 説明だけを更新します。別の **手動でレビューされる** 戻り経路が、Linear のステータス / 優先度 / レーンをプランのフロントマターに引き戻します。

**背景。** 小さなチームには、プロジェクトトラッカーと喧嘩しない、ツールに依存しないレビュー可能なプランニング状態が必要で、自律的なエージェントループには、プラン状態を読み書きできる安定した場所が必要です。

**根拠。** フィールドの所有権を分けることで 2 つのシステムを衝突なく保ちます。markdown がコンテンツを所有し、Linear がトリアージ状態を所有するので、プッシュが人間のトリアージを上書きすることはありません。`dropped/` レーンは、キャンセルされたプランを同期パイプラインから外して再出現しないようにします(許可する例外のガードレール / LAV-5 が却下されたときに作られました)。プラン内の古いフロントマターはステータスではなくドキュメントのバグです。フォルダーが勝ち、「Backlog」のフロントマターにもかかわらずコードが機能の出荷を示している場合(例: アカウント削除)はコードが勝ちます。

**ステータス。** **採用**(`scripts/sync-plans-to-linear.mjs`、`.github/workflows/sync-plans.yml`;`dropped/` レーンを使用中)。

## 11. リポジトリ分割 + クライアントのコピーレフトなオープンソース化 {#11-repo-split-copyleft-open-source-of-the-client}

**決定。** モノレポをコンポーネントごとのリポジトリ(`lavasec-ios`、`-android`、`-web`、`-infra`、`-doc`、`-runner`)に分割し、Mullvad / ProtonVPN のコピーレフトの先例にならって、**ファーストパーティのクライアントを Apache-2.0 ではなく AGPL-3.0 でオープンソース化** する。

**背景。** コンポーネントごとの開発と、クライアントのオープンソース化です。ライセンスの問題は、競合がクライアントをフォークしてクローズ化し、価格で安売りできてしまうのか、という点でした。

**根拠。** コピーレフトは派生物をオープンのままに強制するので、クライアントのクローズドなフォークを防ぎます。「クライアントは公開、バックエンド / 運用は非公開」という姿勢で、バックエンド・法務・運用は非公開のままにします。素の GPL-3.0 ではなく AGPL-3.0 を選んだのは、ネットワーク利用の抜け穴を塞ぐためです。よく知られた GPL と App Store の配布の緊張関係は、Lava 自身が自社の著作権のもとで App Store バイナリの配布者になることで対処しています。

**ステータス。** **採用**。リポジトリ分割は **完了** しています。各コンポーネントはそれぞれのリポジトリにあり、公開の `lavasec-ios` クライアントはタグ v0.4.0、加えて Android、マーケティングサイト、バックエンド / インフラ、ドキュメント、CI / リリースパイプラインのための別々のリポジトリがあります。そして `lavasec-ios` の `README.md` の「Repository layout」セクションは、そのリポジトリのコンポーネントごとの内容(`LavaSecApp/`、`LavaSecTunnel/`、`LavaSecWidget/`、`Shared/`、`Sources/`、`Tests/`)だけを列挙し、インフラは別の非公開リポジトリにあると注記しています。クライアントは **AGPL-3.0** でオープンソース化されています。`lavasec-ios` の `LICENSE` は GNU Affero General Public License v3 で、`README.md` には AGPL-3.0 のバッジが付いています。

## 付録 — そのほかに記録された撤回と却下 {#appendix-other-recorded-reversals-and-rejections}

これらは小さめですが、記録された方向転換を伴う本物の決定でした。網羅のために挙げておきます。

| 決定 | 根拠 | ステータス |
|---|---|---|
| カスタム DNS の無料 vs 有料 | マネタイズの位置づけ;一時は無料で許可したが、有料のみに戻した | 有料のみに **撤回** |
| メール / パスワードのサインイン | パスワードを自前で持つと、リセット / MFA / ロックアウト / 漏洩 / 乗っ取りの負担が増える一方で、Apple + Google で十分;復元をバイパスするとゼロ知識が壊れる | **撤回** / 出荷せず(Apple + Google のみ) |
| 許可する例外のガードレール(LAV-5) | ガードレールの優先順位は、もっとシンプルなフィルターリスト編集の刷新で出荷した;支払いが高信頼度の脅威ガードレールをバイパスしてはならない | **撤回**(`dropped/` レーンを作成) |
| TestFlight のブランチ昇格ロックダウン | 当初のロックダウンを再考し、予定されているオープンソース化後のランナーロックダウンに置き換えた | **撤回**、バックログのプランに置き換え |
| アプリ↔拡張機能のコントロールチャンネル | `sendProviderMessage`(`NETunnelProviderSession`)が **唯一のアプリ→トンネルのコントロールパス** で、型付けされたリビジョン付きの状態を運び、権威をもって拡張機能のランループを駆動します。以前の拡張機能側の `CFNotificationCenter` オブザーバーは実機で確実に発火せず、**削除** されました(ソース内省テストで不在をアサート)。Darwin 通知は **トンネル→アプリ** の方向にだけ、ヘルス変更の合図として残っています。 | **採用**(provider-message が唯一のアプリ→トンネルのコントロール;Darwin はトンネル→アプリのヘルスのみ) |

> 全体を通して参照される横断的な安全不変条件: 支払いは、ハッシュ検証された、許可不能な **脅威ガードレール** を決してバイパスしません。決定の優先順位は **脅威ガードレール > ローカル許可リスト(許可する例外)> ブロックリスト > デフォルト許可** です。
